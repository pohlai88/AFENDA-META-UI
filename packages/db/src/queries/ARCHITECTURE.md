# Database Access Layer — Architecture

> **Status:** Enforced access surface (read layer + mechanical soft-archive)  
> **Package:** `@afenda/db` — `src/queries/`  
> **ORM:** Drizzle ORM  
> **Scope:** **ERP domains only** — `hr`, `sales`, `inventory`, `accounting`, `purchasing` (see CI allowlist in `tools/ci-gate/db-access-layer` when present). Platform namespaces (`core`, `security`, `meta`, `reference`) are out of scope for generated access modules.

---

## Purpose

This directory defines the **canonical database access surface for reads** (CQRS-lite, enterprise minimal). It exists to ensure:

- **Consistent tenant scoping** (RLS plus explicit `tenantId` where applicable — defense in depth)
- **Consistent soft-delete handling** for tables that define `deletedAt`
- **Stable, reusable access contracts** so applications do not duplicate the same Drizzle shapes
- A clear split between **generated baseline** code and **human-owned** reporting and analytics

This is **not** a convenience dumping ground. It is a **consistency and safety boundary**.

This layer is read-first. The only permitted write is soft-archive (`archiveX`) as a mechanical operation aligned with soft-delete columns. All other mutations belong to the command layer.

## Position in the stack

```
apps/api, workers, jobs
        │
        ▼
@afenda/db  src/queries/   ← ACCESS SURFACE (this layer)
        │
        ▼
@afenda/db  src/schema/    ← tables, relations
@afenda/db  src/rls-policies/         ← tenant policies
@afenda/db  src/pg-session/            ← Postgres GUC session context
        │
        ▼
@afenda/meta-types         ← optional type contracts; see below
```

**Consumers** use this layer for reusable reads, safety-critical access patterns, and reporting primitives. **Orchestration** (HTTP, authz branching, DTO mapping) stays in applications.

## Core principles

### 1. Read-first

- **SELECT** and query builders for reads belong here.  
- **Reporting / aggregates / domain read models** belong in human-owned modules (e.g. `workforceReports.ts`).  
- **Inserts, updates, hard deletes** do not belong here — except the single mechanical exception below.

### 2. Deterministic access

Generated and hand-written modules in this layer should be **explicit, predictable, and consistent** across domains (naming, tenant filters, soft-delete semantics).

### 3. No business logic (in the baseline)

**Allowed in access/report modules:** filtering, joins within a domain, aggregation for read shapes.  
**Forbidden:** business rules that belong in policy/API, conditional workflows, authorization branching, side effects outside the database client.

### 4. Tenant safety

All access must respect **RLS** (session context) where the connection is tenant-scoped, and use **explicit `tenantId` predicates** in generated baselines where the schema defines `tenantId` — especially for jobs or service paths that may bypass session.

## Generated vs human code

### Generated access modules (baseline)

- **Naming:** `*.access.ts` (one per **schema source file** under an ERP domain, e.g. `workforceStrategy.ts` → `workforceStrategy.access.ts`).
- **Marker:** files are marked `@generated` and may be regenerated by CI/tooling (v2); v1 may use placeholders only.
- **Responsibilities:** tenant-aware patterns, soft-delete-aware **Safe** / **Active** / **All** lists, and **`archiveX`** only when the table has a soft-delete column (see naming table).

### Graph-validation guard (optional, hybrid)

- Each baseline exports **`*Guarded`** variants (e.g. `getXByIdSafeGuarded`) that call `assertGraphGuardrailAllowsRead()` from [`_shared/graph-guardrail.ts`](./_shared/graph-guardrail.ts).
- **Default:** if `GRAPH_VALIDATION_POLICY_JSON` is **unset**, guarded functions behave like unguarded (no overhead).
- **When set:** inject the `policy` object (or full v1 report JSON) from `pnpm graph-validation report --format=json`. **Hard block** when `policy.isSecurityBlocking === true` or `policy.decision.action === "BLOCK"`. Staleness: `GRAPH_VALIDATION_POLICY_STALE_MODE` — see [`../graph-validation/POLICY_LIFECYCLE.md`](../graph-validation/POLICY_LIFECYCLE.md).
- **Graded policy:** `decision.severity` (P0–P3) and `decision.action` (`BLOCK` | `WARN` | `ALLOW`); use `graphGuardHasOperationalSignal()` for non-blocking UX banners.
- **Telemetry:** `setGraphGuardrailTelemetrySink()` in [`_shared/guard-telemetry.ts`](./_shared/guard-telemetry.ts); feedback process in [`../graph-validation/FEEDBACK_LOOP.md`](../graph-validation/FEEDBACK_LOOP.md).
- **Feature flag (app layer):** enable `*Guarded` call sites per route or service before making them the default; keep env unset in environments that should not enforce policy.
- Promotion gates: [`src/graph-validation/GUARDRAIL_PROMOTION.md`](../graph-validation/GUARDRAIL_PROMOTION.md).

### Drizzle usage (data-plane rules)

- **Relational Queries API (`db.query`):** use for UI/API nested shapes and **human-owned** report modules when you need one SQL round-trip; **do not** adopt in generated `*.access.ts` unless the generator explicitly supports it; **never** in `graph-validation` offline engine (see [Drizzle RQB](https://orm.drizzle.team/docs/rqb)).
- **CTEs / subqueries:** prefer for multi-step reports (deterministic, explainable) — see `.agents/skills/drizzle-orm/references/query-patterns.md`.
- **`db.batch`:** use when multiple independent statements must run together; prefer a single relational or CTE query when possible — document the choice in the module.
- **`db.transaction`:** only for **truth-consistency** (e.g. set RLS/GUC then read same snapshot via [`pg-session`](../pg-session/)); avoid extra transactions for latency/pool pressure.
- **Read replicas / serverless pooling:** see `.agents/skills/drizzle-orm/references/performance.md` and official [Performance](https://orm.drizzle.team/docs/performance) / driver docs.
- **Prepared patterns:** optional on proven hot paths after measurement.

### Truth surfaces

High-value reads may adopt the `TruthSurface<T>` pattern — see [`../graph-validation/truth-surface.ts`](../graph-validation/truth-surface.ts). This is a **contract/documentation** aid; offline graph-validation remains authoritative.

### Human-owned query modules

- **Examples:** `workforceReports.ts`, `pipelineAnalytics.ts`.
- **Responsibilities:** reporting, richer joins, metrics, domain read models.
- **Rules:** never overwritten by generators; encode **meaning**, not only mechanical access.

## Naming conventions (do not mix semantics)

| Function         | Rule                                      |
| ---------------- | ----------------------------------------- |
| `getXByIdSafe`   | Tenant-scoped; **excludes** soft-deleted rows |
| `listXActive`    | **Excludes** soft-deleted rows            |
| `listXAll`       | **Includes** soft-deleted rows            |
| `archiveX`       | Soft-delete only (mechanical update)      |

Tables **without** `deletedAt` do not get `*Safe` / `*Active` / `*All` / `archiveX` variants from the generator; use tenant-scoped access patterns appropriate to that table (v2 generator detects columns; do not assume).

## `_shared/` constraints

**Allowed:** pagination helpers, generic date/cursor utilities, non-domain SQL fragments.  
**Forbidden:** domain-specific joins, business semantics, HR/Sales-only rules.

---

## When applications use this layer

Applications **should** use this layer for:

- Reused read paths and reporting primitives  
- Access patterns where tenant and soft-delete correctness must stay consistent  

Applications **may** use raw Drizzle inline for one-off or early prototyping; **promote** shared logic here when a second consumer appears or safety demands it.

## Generator and CI (reference)

- **Check:** `pnpm ci:gate:db-access` — each allowlisted schema module must have a non-empty sibling `queries/<domain>/<basename>.access.ts`. No AST validation of function bodies.  
- **Placeholder fix:** `pnpm ci:gate:db-access:fix` creates stubs where missing.  
- **Full emit:** `pnpm ci:gate:db-access:generate` (`--fix --generate --force`) emits `getXByIdSafe`, `listXActive`, `listXAll`, and `archiveX` (when `softDeleteColumns` is present), refreshes `queries/<domain>/index.ts` barrels, and overwrites only files containing `@generated`.  
- **ID types:** use `(typeof table.$inferSelect)["id"]` — never hardcode numeric vs string IDs.  
- **Database type:** `import type { Database } from "../../drizzle/db.js"` (adjust depth); do **not** use `NodePgDatabase`.  
- **Soft-delete:** detected via schema extractor (`hasSoftDeleteColumns`); omit Safe/Active/All/archive when absent.

See [`tools/ci-gate/db-access-layer/README.md`](../../../tools/ci-gate/db-access-layer/README.md).

---

## Alignment with `@afenda/meta-types`

Use `import type { … }` from `@afenda/meta-types` for shared parameter shapes (e.g. session fields) where useful. Do not create circular dependencies; meta-types must not import `@afenda/db`. Business invariants stay in policy/API/truth layers — not in this access surface.

---

## Out of scope

- HTTP, cookies, status codes  
- DTO shaping for external APIs (map in the app)  
- Command orchestration, general CRUD services (except mechanical `archiveX` per Option A)  
- Platform schema domains in generated `*.access.ts`

## Summary

- **`src/queries/`** is the **deterministic database read access surface** (plus mechanical soft-archive only).  
- **`*.access.ts`** = generated baseline; other modules = human projections and reports.  
- **ERP-only**; **tenant + soft-delete** semantics are explicit and named.  
- **Apps** orchestrate; **this layer** stays Drizzle-native and consistent with RLS/session expectations.

## Placement Rules (Authoritative)

- Put each table in the narrowest bounded context that owns its business lifecycle.
- If two contexts are plausible, choose the one that owns mutation and invariants, not just read usage.
- Keep cross-context references as explicit FKs and mirror each new edge in `_relations.ts`.
- Do not create generic file names (`tables.*`, `sales.*`) for domain modules.

## Mandatory Patterns

### 1) Tenant Isolation

- Tenant-scoped tables must include `tenantId`.
- Keep tenant-leading indexes for tenant filters.
- Apply both RLS helpers in each table block:
  - `...tenantIsolationPolicies("<table_name>")`
  - `serviceBypassPolicy("<table_name>")`

### 2) Foreign Keys

- Every FK must define explicit `onDelete` and `onUpdate`.
- For intra-sales edges, update `_relations.ts` in the same PR.
- For external references (`core`, `reference`, `security`), keep imports explicit and minimal.

### 3) Check Constraints

Use `check()` for domain invariants such as:

- non-negative totals and amounts
- bounded percentages and discounts
- date/effective range ordering
- workflow consistency constraints

### 4) Soft-Delete Aware Uniqueness

When a table uses `...softDeleteColumns`, prefer partial uniques:

- `uniqueIndex(...).where(sql\`${table.deletedAt} IS NULL\`)`

### 5) Enum and Contract Discipline

- Enum tuples + pgEnum + Zod + exported type must remain aligned in `_enums.ts`.
- Domain files should export select/insert/update schemas and inferred row types for owned tables.
- If update schema is intentionally omitted (append-only table), document the exception in module docs or PR notes.

### 6) Shared Column and Validation Helpers

Use shared helpers; do not re-implement standard columns/scalars:

- `timestampColumns`
- `auditColumns`
- `softDeleteColumns`
- `nameColumn`
- shared ID/scalar schemas from `_zodShared.ts`

## Relations Catalog Requirement

`_relations.ts` is mandatory metadata and must track FK topology.

- Any FK add/remove/change -> update `_relations.ts`.
- `RELATIONS_DRIFT` gate must pass with zero sales-domain errors.

## Documentation-First Rule

Before structural schema changes:

1. Update `README.md` and `ARCHITECTURE.md` for file-map/DAG/ownership impact.
2. Apply code changes in bounded-context files.
3. Re-run validation gates and record results.

No trial-and-error file naming or exploratory generic modules.
